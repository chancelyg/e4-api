# AGENTS.md

## Purpose

- This repository is a small diary application with a Go backend and a Svelte 5 frontend.
- The backend serves JSON APIs and embeds the built frontend from `pkg/dist` into the final Go binary.
- The frontend source lives in `web/`; production assets are built into `web/dist/` and then copied to `pkg/dist/` by `build.sh`.
- Prefer small, targeted changes that preserve the current architecture and naming patterns.

## Repo Layout

- `main.go`: application entrypoint, Echo wiring, middleware, API routes, SPA fallback.
- `internal/config`: config loading via Viper and default settings.
- `internal/db`: GORM + SQLite initialization.
- `internal/handlers`: HTTP handlers for auth, diary, goals, json_store, and common endpoints.
- `internal/middleware`: session-based auth middleware.
- `internal/models`: GORM models and JSON field definitions.
- `pkg/embed.go`: embeds `pkg/dist` into the Go binary.
- `pkg/dist`: generated frontend build copied from `web/dist`.
- `web/src`: Svelte 5 app source.
- `web/dist`: generated frontend output.
- `web/.svelte-kit`: generated SvelteKit files.

## Commands

### Setup

- Install backend tooling: Go `1.25+`.
- Install frontend tooling: Node/npm compatible with SvelteKit 2 and Vite 7.
- Root and frontend dependencies are not managed by a single workspace tool.

### Development

- Full-stack dev helper: `./dev.sh`
- Backend only: `go run main.go`
- Frontend only: `cd web && npm install && npm run dev`

### Build

- Full production build: `./build.sh`
- Frontend build only: `cd web && npm install && npm run build`
- Backend binary only after frontend assets exist in `pkg/dist`: `go build -ldflags="-s -w" -o e4-api main.go`

### Lint / Type Check

- There is no dedicated ESLint, Prettier, or golangci-lint config in this repo.
- Frontend static check: `cd web && npm run check`
- Frontend watch check: `cd web && npm run check:watch`
- Go formatting check by convention: `gofmt -w <files>`
- Go vet is not scripted, but reasonable for backend changes: `go vet ./...`

### Tests

- Run all Go tests: `go test ./...`
- Verbose Go tests: `go test ./... -v`
- Run one package: `go test ./internal/handlers -v`
- Run one test: `go test ./internal/handlers -run TestLoginSuccess -v`
- Run one test in another package: `go test ./pkg -run TestEmbeddedFiles -v`
- Run subtests by regex when useful: `go test ./internal/handlers -run 'TestCalculateStats/single diary' -v`
- Frontend test runner is not configured right now.

## Verification Expectations

- For backend-only changes, at minimum run the relevant `go test` target.
- For frontend-only changes, at minimum run `cd web && npm run check`.
- For cross-stack changes, prefer running both `go test ./...` and `cd web && npm run check`.
- If you change embedded asset behavior, also run the frontend build flow and ensure `pkg/dist` is refreshed.

## Generated Files

- Do not hand-edit `web/dist/`; regenerate it with `cd web && npm run build`.
- Do not hand-edit `web/.svelte-kit/`; it is generated by SvelteKit.
- Do not hand-edit `pkg/dist/`; it is copied from `web/dist/` during `./build.sh`.
- Ignore `web/node_modules/` for code changes.
- Treat SQLite DB files under `data/` as runtime data, not source.

## Backend Style

- Follow standard Go formatting with `gofmt`; imports are grouped automatically by Go tooling.
- Keep packages small and focused: config in `internal/config`, HTTP logic in `internal/handlers`, persistence wiring in `internal/db`.
- Use constructor helpers like `NewAuthHandler`, `NewDiaryHandler`, and `NewAuthMiddleware` for setup.
- Exported types and functions use PascalCase; unexported helpers use camelCase.
- Handler methods use the shape `func (h *Type) Method(c echo.Context) error`.
- Request structs are declared near the handler that uses them.
- Response payloads are usually returned with inline `map[string]...` for simple cases or small dedicated structs for stable shapes.

## Backend Imports

- Keep standard library imports first, then local `e4-api/...` imports, then third-party imports.
- Use aliases only when needed to disambiguate, such as Echo middleware in `main.go`.
- Avoid unnecessary imports; this codebase keeps import lists tight.

## Backend Types and Data Shapes

- JSON field names are snake_case, matching current API contracts like `create_date`, `is_logged_in`, and `max_consecutive_days`.
- GORM models use explicit struct tags for both database and JSON behavior.
- Config structs use `mapstructure` tags for Viper unmarshalling.
- Prefer concrete structs over `map[string]any` unless the response is tiny and ad hoc.
- Keep date strings in `YYYY-MM-DD` format where existing APIs expect strings instead of `time.Time` JSON values.

## Backend Error Handling

- Return errors to clients with `c.JSON(status, map[string]string{"error": "..."})`.
- Keep error messages user-facing and consistent with the existing Chinese-language API responses.
- Validate request parsing early with `c.Bind` checks.
- Validate path params and dates before DB operations.
- Distinguish `gorm.ErrRecordNotFound` from generic DB failures.
- Use `log.Fatalf` in startup code where the app cannot continue.
- Do not ignore important errors in new code; existing unchecked calls are not a style target to expand.

## Backend Testing Patterns

- Tests live beside the package under test in `*_test.go` files.
- Current tests use `testing` plus `stretchr/testify` `assert`/`require`.
- Prefer table-driven tests for pure logic, as in `TestCalculateStats`.
- For handler tests, construct an Echo instance, request, recorder, and context explicitly.
- Keep test setup helpers small and package-local.

## Frontend Style

- The frontend is Svelte 5 with runes enabled for project files.
- Use `<script lang="ts">` and keep types explicit when data shapes are stable.
- Local component state uses `$state(...)` rather than legacy store patterns.
- Shared auth state is wrapped in a custom factory in `web/src/lib/stores.svelte.ts`.
- Route components generally fetch on `onMount` and redirect with `goto(...)` when auth is missing.
- Keep API access centralized in `web/src/lib/api.ts`.

## Frontend Imports and Modules

- Use `$lib/...` and `$app/...` aliases instead of long relative paths when available.
- Keep imports grouped simply; the codebase does not use a strict import-sorting tool.
- Export shared API types from `web/src/lib/api.ts` and consume them in route files.

## Frontend Naming and Conventions

- Use camelCase for local variables and functions.
- Type/interface names use PascalCase.
- Match backend field names in API types even when they are snake_case.
- Event handlers are typically named `handleX`, loaders `loadX`, and formatters `formatX`.
- Use clear booleans like `isLoading`, `isSaving`, `isEditing`, `showDeleteConfirm`.

## Frontend Error Handling

- Shared fetch wrapper throws `Error` with API-provided messages when possible.
- UI code usually catches errors locally and either sets an `error` state or logs to console.
- Preserve existing user-facing language; current UI messages are Chinese.
- For forms, clear prior errors before retrying.

## Frontend Styling

- Styling is plain CSS in `web/src/app.css`, not Tailwind utilities generated from a framework config.
- Many class names resemble utility classes, but they are custom classes defined in `app.css`.
- Reuse existing design tokens and utility-style classes before inventing new one-off patterns.
- Keep visual additions aligned with the current simple blue/neutral palette and card-based layout.
- Preserve responsive behavior already implied by width, flex, and max-width utility classes.

## Architecture Notes

- The Go server serves API routes under `/api` and static SPA content for everything else.
- Auth uses signed session cookies (HMAC-SHA256); revocation records are persisted to SQLite.
- CORS is currently limited to `http://localhost:5173` for development.
- `pkg/embed.go` requires built frontend assets to exist before packaging the backend binary.
